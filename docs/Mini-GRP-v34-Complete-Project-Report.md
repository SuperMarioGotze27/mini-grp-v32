# Mini-GRP v3.4 完整项目报告

> 项目性质：面试衍生的量化研究与工程实践项目
> 报告版本：v3.4.0
> 报告日期：2026-06-12
> 代码仓库：github.com/SuperMarioGotze27/mini-grp-v32
> 当前定位：可解释的多因子研究系统、受治理的机器学习实验平台与可部署的股票筛选应用
> 重要声明：本项目不是自动交易系统，报告中的短样本结果不构成投资建议或历史业绩陈述

## 执行摘要

Mini-GRP 起源于一次投资研究相关面试。项目目标不是复刻 Principal 内部的 Global Research Platform，而是把公开材料中“系统覆盖大量公司、相对行业同业比较、量化筛选支持基本面研究”的思路，转化为一个个人可以完成、解释、运行和持续改进的软件项目。

项目最初是 A 股多因子选股原型，随后经历了数据源扩展、评分模型扩展、机器学习实验、Streamlit 展示、回测和部署等多轮迭代。审查发现，v3.2 虽然已经具有较完整的产品外观和文档，但远程仓库、真实数据、回测、机器学习和部署之间仍然存在断层：

- 远程仓库曾缺少本地完整模块。
- Streamlit 与核心代码存在重复逻辑。
- 真实数据失败时可能回退到模拟数据。
- 原回测主要用于演示，并不能代表真实历史策略。
- ML 模块虽然存在，却没有真实历史标签、模型注册和生产推理闭环。
- Streamlit Community Cloud 可以运行网页，但不适合承担长期历史采集和定时训练。

v3.3 先解决可靠性和可审计性问题；v3.4 再完成研究闭环。当前版本新增了：

1. Tushare 历史月末快照采集。
2. SQL 持久化研究数据库。
3. 未来 20 个交易日收益标签。
4. 扩展窗口 Walk-forward 机器学习验证。
5. 候选模型、批准模型和模型版本注册。
6. 受上限约束的 ML Overlay 推理路径。
7. 基于真实快照的线性基准回测。
8. Streamlit 模型状态与研究回测页面。
9. Google Cloud Run、Cloud Run Job、Cloud SQL 和 Secret Manager 部署方案。

真实演练也给出了克制而重要的结果。截至 2026-06-12，使用最近 14 个交易月、每期最多 300 只大市值 A 股进行验证，获得 4,200 条快照记录，其中 12 个月具备完整未来收益标签，共 3,593 条训练样本。Gradient Boosting 虽然在候选模型中相对更好，但样本外平均 Rank IC 为 -0.046，未达到审批门槛，因此系统没有启用 ML 排序。

同期线性 Top-20 快照回测年化收益约 14.3%，等权股票池约 29.0%，年化超额约 -14.7%，信息比率约 -1.40。这个结果说明当前因子组合在该短样本和股票池定义下没有跑赢基准。v3.4 的价值不是制造“漂亮结果”，而是建立了一个能够真实发现模型无效、并阻止无效模型上线的研究系统。

## 1. 项目起点与问题定义

### 1.1 面试触发

项目最初希望证明三项能力可以结合：

- 理解基本面和量化投资研究逻辑。
- 把研究逻辑转化为字段、因子、模型和程序。
- 把程序组织成可复现、可演示、可部署的工程项目。

核心研究问题是：

> 在有限数据、有限算力和有限开发时间下，如何构建一个可解释、可扩展，并能支持人工研究的股票初筛系统？

这个问题进一步拆分为：

1. 如何统一不同数据源和市场的字段。
2. 如何处理异常值、缺失值、因子方向和量纲。
3. 如何在部分维度不可用时维持一致的评分逻辑。
4. 如何构造不使用未来信息的历史训练样本。
5. 如何验证机器学习确实比简单模型增加了信息。
6. 如何让网页展示、批处理、数据库和部署使用同一套核心逻辑。

### 1.2 对 GRP 概念的适度映射

| 公开概念 | Mini-GRP 实现 | 主动保留的边界 |
|---|---|---|
| 扩大研究覆盖 | 对统一股票池批量计算因子 | 不声称达到机构级全球覆盖 |
| 同业相对比较 | 市场内标准化、行业内排名 | 尚未建立完整行业专属模型 |
| 量化筛选支持研究 | Top-N、维度拆解、数据来源标记 | 不替代分析师尽调和投资决策 |
| 研究过程系统化 | 快照、验证、模型注册、版本记录 | 不把短样本结果包装成稳定 Alpha |

更准确的表述是“受 GRP 公开理念启发的个人研究工程项目”，而不是“复制 GRP”。

## 3. 当前系统架构

```text
                         +----------------------+
                         | Tushare live endpoint |
                         +----------+-----------+
                                    |
              +---------------------+----------------------+
              |                                            |
              v                                            v
     实时股票截面                                  月末历史快照采集
              |                                            |
              v                                            v
      因子计算与线性评分                         SQL research_snapshots
              |                                            |
              |                           +----------------+----------------+
              |                           |                                 |
              |                           v                                 v
              |                    真实线性基准回测                  Walk-forward ML
              |                                                             |
              |                                                     model_registry
              |                                                             |
              +--------------------------+----------------------------------+
                                         |
                                  审批后的 ML Overlay
                                         |
                                         v
                              Streamlit / CSV / 模型监控
```

### 3.1 分层职责

| 层级 | 目录 | 主要职责 |
|---|---|---|
| 数据适配 | `data/` | 在线数据获取、字段映射、缓存、来源标记 |
| 因子与线性评分 | `core/` | 去极值、标准化、方向调整、维度聚合、行业排名 |
| 研究持久化 | `research/storage.py` | SQL 快照和模型制品 |
| 历史采集 | `research/collector.py` | 月末截面、动量价格、未来收益标签 |
| ML 训练 | `research/trainer.py` | 扩展窗口验证、模型选择和审批 |
| 推理 | `research/inference.py` | 批准模型加载、ML 分数与线性分数混合 |
| 回测 | `research/backtest.py` | 真实快照线性组合、成本、基准和指标 |
| 展示 | `streamlit_app.py` | 选股、模型注册、回测、方法说明 |
| 部署 | `deploy/gcp/` | Cloud Run 服务和 Job |

### 3.2 两类运行路径

**实时筛选路径**

1. 获取最新可用股票截面。
2. 计算当前因子和线性综合分。
3. 如果用户选择 ML 且数据库存在批准模型，则加载模型。
4. 将 ML 横截面预测转换为 0-100 百分位。
5. 按受限权重混合并输出 Top-N。

**离线研究路径**

1. 获取交易日历和月末交易日。
2. 采集估值、基本面、历史价格。
3. 计算 1M、3M、12M 动量。
4. 使用未来第 20 个交易日价格构造标签。
5. 写入 SQL。
6. 使用早期月份训练、较晚月份验证。
7. 只有通过门槛的模型才能成为实时推理模型。

## 4. 数据层与点时性

### 4.1 当前真实 A 股字段

| 维度 | 当前可获得字段 |
|---|---|
| Value | PE TTM、PB、PS TTM、股息率 |
| Quality | 毛利率、净利率 |
| Growth | 营收同比、利润同比 |
| Momentum | 1M、3M、12M 收益 |

当前真实路径共使用 11 个因子。EV/EBITDA、ROE、ROA、杠杆、FCF Yield 和分析师预期因子需要进一步的数据源支持。

### 4.2 月末快照

`research_snapshots` 的联合主键是 `snapshot_date + code`。每条记录包含：

- 股票代码、名称、行业和市场。
- 11 个原始因子。
- 标签日期。
- 未来 20 个交易日收益。
- 写入时间。

采集任务采用增量逻辑：

- 已完整采集的旧月份跳过。
- 最新两个月重新抓取，以便未来标签成熟后补齐。
- 若旧月份的股票数量低于新的 `max_stocks`，则自动重新采集。

### 4.3 点时性边界

系统避免在训练中直接使用标签期之后的信息，但仍需承认供应商历史接口的边界：

- `bak_basic` 历史字段不一定等价于严格的财报发布日期点时数据库。
- 退市股票和历史指数成分可能带来幸存者偏差。
- 财务数据可能包含后续修订。
- 当前股票池按总市值排序，研究结论偏向大市值股票。

因此，现有真实回测比合成回测更有意义，但仍不应称为机构级 point-in-time 回测。

## 5. 因子处理与线性模型

### 5.1 处理顺序

1. 数值类型转换和可用性检查。
2. 缺失值分组填充。
3. 分位数去极值。
4. 市场内 Z-Score 标准化。
5. 按经济方向调整正负号。
6. 维度内等权聚合。
7. 按可用维度重新归一化政策权重。
8. 转换为横截面百分位。
9. 计算行业内排名。

### 5.2 政策权重

| 维度 | 权重 |
|---|---:|
| Value | 25% |
| Quality | 25% |
| Growth | 15% |
| Momentum | 15% |
| Expectation | 20% |

若 Expectation 没有真实数据，该维度被排除，其余维度按比例归一化。系统不再通过随机数维持表面上的五维完整性。

### 5.3 线性模型的角色

线性模型承担三个职责：

- 提供清晰可解释的基础排序。
- 在 ML 不通过验证时作为安全回退。
- 为 ML 提供需要被增量改善的基准。

这比直接把所有决策交给黑盒模型更适合当前样本规模和项目定位。

## 6. 机器学习闭环

### 6.1 训练目标

原始未来收益容易受极端值和市场环境影响。v3.4 将每个月的未来 20 日收益转换为横截面百分位：

`target_rank = percentile_rank(forward_return) * 100`

模型学习的是股票在同月截面中的相对位置，而不是直接预测精确收益率。

### 6.2 模型候选

当前稳定生产候选只有两类：

- Ridge：低方差、可作为正则化线性基准。
- Gradient Boosting：捕捉有限的非线性与交互。

仓库中原有 XGBoost、SHAP、HMM 和多臂老虎机模块仍保留为高级研究工具，但不自动进入生产路径。

### 6.3 Walk-forward 验证

对每个验证月：

1. 只使用该月之前的所有月份训练。
2. 在该月截面预测。
3. 计算预测排序与未来收益的 Rank IC。
4. 计算预测 Top 20% 与 Bottom 20% 的未来收益差。
5. 向后滚动到下一验证月。

这种扩展窗口方式比随机拆分更符合金融时间序列约束。

### 6.4 审批门槛

模型只有在以下条件同时满足时才标记为 `approved`：

- 至少 3 个样本外验证折。
- 平均 Rank IC 大于 0。
- 平均 Top-Bottom 收益差大于 0。

否则模型只保存为 `candidate`。实时页面只允许加载批准模型。

### 6.5 Overlay 规则

批准模型的最终得分为：

`final_score = (1 - λ) * linear_score + λ * ml_score`

默认 `λ = 0.15`，代码硬性限制不超过 0.30。ML 即使通过验证，也不能完全覆盖线性研究逻辑。

## 7. 真实验证结果

### 7.1 数据演练

2026-06-12 使用真实 Tushare 代理接口完成：

| 项目 | 结果 |
|---|---:|
| 快照范围 | 2025-05-30 至 2026-06-12 |
| 月末快照数 | 14 |
| 数据库记录 | 4,200 |
| 有完整标签月份 | 12 |
| 训练样本 | 3,593 |
| 每月最大股票数 | 300 |
| 标签窗口 | 未来 20 个交易日 |

### 7.2 ML 结果

| 模型 | 平均样本外 Rank IC | 平均 Top-Bottom Spread | 状态 |
|---|---:|---:|---|
| Ridge | -0.073 | -0.35% | 未批准 |
| Gradient Boosting | -0.046 | -1.94% | 候选，未批准 |

Gradient Boosting 被选为相对较好的候选，但两个审批指标未同时为正，因此没有上线。

该结果说明：

- 12 个标签月份仍然太短。
- 当前大市值股票池和因子方向未产生稳定样本外信号。
- 非线性模型没有自动创造 Alpha。
- 模型审批机制按预期阻止了负 IC 模型进入实时选股。

### 7.3 线性真实快照回测

参数：

- 每月 Top-20 等权。
- 单边交易成本 0.10%。
- 基准为当月可用股票池等权收益。
- 12 个持有期。

| 指标 | 结果 |
|---|---:|
| 策略年化收益 | 14.31% |
| 基准年化收益 | 29.01% |
| 年化超额 | -14.70% |
| 最大回撤 | -5.49% |
| 信息比率 | -1.40 |
| 平均换手率 | 41.25% |

绝对收益为正不能掩盖相对结果：在该样本中，Top-20 没有跑赢等权股票池。

### 7.4 对结果的解释

可能原因包括：

- 样本期仅约一年，市场状态单一。
- 股票池按总市值优先，金融和大盘股暴露较强。
- Value 与 Quality 的当前数据覆盖有限。
- 没有进行行业权重约束和组合风险控制。
- 没有分析师预期、现金流和更严格的盈利质量数据。
- 固定政策权重未根据 A 股样本重新校准。

下一阶段不应首先“调参把结果变正”，而应扩大历史、改善点时数据、分析行业暴露并建立朴素基线比较。

## 8. Streamlit 产品形态

v3.4 页面分为四个区域。

### 8.1 Screening

- Demo 或 Research 数据模式。
- CN/US 市场入口。
- 线性基准或 Approved ML Overlay。
- 股票池大小和 Top-N。
- 数据来源、因子覆盖率和模型版本。
- Top-N 表格、雷达图、行业分布和 CSV 下载。

当没有批准模型时，选择 ML 模式会明确报错，不会偷偷使用候选模型。

### 8.2 Model registry

- 快照日期数、记录数、模型数和最新日期。
- 批准模型版本和训练范围。
- Walk-forward Rank IC 和 Top-Bottom Spread。
- 每个验证月的样本外结果。
- 模型特征和因子诊断。

### 8.3 Backtest

- Stored research snapshots：真实快照线性基准回测。
- Synthetic pipeline check：只用于流程和会计逻辑回归测试。

页面明确区分两者，避免把合成结果解释为投资证据。

### 8.4 Methodology

方法页说明数据、因子、模型审批、ML 权重上限和回测边界。

## 9. Google Cloud Run 部署

### 9.1 为什么不把全部任务放在 Streamlit Community Cloud

Streamlit Community Cloud 适合轻量网页，但历史采集和训练具有不同特征：

- 任务运行时间更长。
- 需要可靠数据库。
- 需要定时执行。
- 需要独立的 CPU、内存和失败重试。

因此 v3.4 采用服务与任务分离。

### 9.2 目标拓扑

| 组件 | 职责 |
|---|---|
| Cloud Run Service | Streamlit、实时筛选、模型读取 |
| Cloud Run Job | 增量采集、训练、模型注册 |
| Cloud SQL PostgreSQL | 快照和模型制品 |
| Secret Manager | Tushare Token、数据库 URL |
| Artifact Registry | 容器镜像 |
| Cloud Scheduler | 每月触发 Cloud Run Job |

### 9.3 容器设计

同一个 Docker 镜像承担两种角色：

- 默认命令启动 Streamlit，并监听 Cloud Run 注入的 `PORT`。
- Job 覆盖命令为 `python -m research.cli pipeline ...`。

这种方式减少了环境漂移，网页和训练任务使用完全相同的代码和依赖版本。

### 9.5 安全原则

## 12. 当前限制

### 12.1 数据

- 财务字段不完全是严格发布日期点时数据。
- 历史股票池存在幸存者偏差风险。
- 分析师预期数据尚未接入。
- 当前真实历史长度不足以评估跨周期稳健性。

### 12.2 模型

- 当前 ML 未通过审批。
- 固定因子权重尚未进行长期校准。
- 没有组合层面的行业、风格、波动率和流动性约束。
- 没有模型漂移、数据漂移和异常值告警。

### 12.3 交易

- 没有真实下单。
- 没有涨跌停、停牌、成交量和冲击成本模拟。
- 没有处理公司行动、可交易性和合规限制。

建议按以下顺序展示：

一个合适的总结表述是：

> 我最初做的是一个能展示多因子思想的选股原型。迭代后，我发现真正困难的不是再加一个模型，而是数据身份、点时性、验证和部署边界。v3.4 的主要成果是建立了真实历史采集、模型审批和部署闭环。真实结果没有支持上线 ML，也没有跑赢基准，因此系统保持线性基准，并把失败结果完整暴露。这让我可以基于证据继续改进，而不是基于展示效果调参。

## 16. 结论

Mini-GRP 已从一个多因子演示原型，演进为具备以下能力的研究系统：

- 真实数据接入。
- 可解释线性评分。
- 历史快照和标签。
- Walk-forward 机器学习验证。
- 模型审批和版本注册。
- 真实快照基准回测。
- Streamlit 交互展示。
- Cloud Run 生产部署结构。

但“完整系统”不等于“已验证策略”。当前最重要的研究结论恰恰是：

- 真实短样本不支持启用 ML。
- 线性 Top-20 没有跑赢等权基准。
- 数据点时性和样本长度仍是主要限制。

因此，v3.4 达成的是工程闭环和研究纪律，而不是对未来收益的保证。它已经适合作为面试项目展示系统设计、量化研究、模型治理和云部署能力；若要进入真实投资流程，还需要更长、更严格的点时数据和组合风险管理。
