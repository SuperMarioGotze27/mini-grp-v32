# Mini-GRP v3.3 完整项目报告

> 项目性质：面试衍生的量化研究与工程实践项目
> 报告版本：v3.3.0
> 报告日期：2026-06-12
> 代码仓库：[SuperMarioGotze27/mini-grp-v32](https://github.com/SuperMarioGotze27/mini-grp-v32)
> 结论定位：可解释的量化初筛系统与可审计的研究框架，不是自动交易系统

## 执行摘要

Mini-GRP 的出发点，是把一次面试中接触到的机构投资研究理念，转化为一个可以运行、解释、验证和持续迭代的个人项目。项目参考了 Principal 历史公开材料中对 Global Research Platform（GRP）的描述：GRP 是一种用于扩大研究覆盖面、提高研究客观性并把分析师注意力集中到较小候选集合的股票排名框架。公开材料还强调了相对区域行业同业进行比较，以及量化筛选与基本面研究结合的思路。

Mini-GRP 没有试图复制机构系统的全部复杂度，而是抽取了一个可交付的核心闭环：统一数据接口、因子处理、五维评分、行业内排名、Top-N 输出、Walk-forward 评估、可视化与部署。它从最初的 A 股 16 因子原型，逐步扩展到 19 因子、预期差维度、跨市场接口、ML 实验模块和 Streamlit 应用。

本次 v3.3 改进的重点不是继续增加功能，而是把项目从“能展示的原型”推进到“行为清楚、结果可追溯、失败不伪装”的研究工程版本。主要成果包括：

1. 恢复并纳入完整模块代码，解决远程仓库只有少量入口文件、无法体现真实工程内容的问题。
2. 修复网页入口语法错误、评分列重复、回测日期错位、错误导入、交易成本单位、随机基准、年化频率和累计超额计算等问题。
3. 将演示数据与真实研究数据严格分离。研究模式禁止静默回退到模拟数据。
4. 为数据增加来源、是否模拟、数据日期、因子覆盖率和预期数据来源等可追溯字段。
5. 将跨市场因子改为市场内标准化，避免不同会计口径和估值水平被直接混合比较。
6. 重建可复现的 Walk-forward 回测核心，输出逐期结果、权益曲线、持仓、指标和运行清单。
7. 新增 7 项自动测试、GitHub Actions、Docker 部署和经过浏览器实际验证的 Streamlit 页面。

当前版本适合用于面试演示、研究流程验证、因子工程学习和后续接入真实点时数据。它不应被包装为已验证的投资策略，也不能把合成数据回测结果解释为历史业绩。

## 1. 项目起点与问题定义

### 1.1 面试触发

项目最早服务于一个非常具体的目标：在面试中证明自己不仅能讨论投资框架，也能把投资思想转换为可运行的软件。传统准备方式通常只能分别展示编程能力或金融知识，而 Mini-GRP 试图展示两者的交集：

- 把抽象的股票研究原则转成字段、因子、评分规则和接口。
- 把海量股票压缩成可以由分析师继续研究的候选列表。
- 为每个候选标的保留价值、质量、增长、动量和预期差的解释路径。
- 把模型限制、数据质量和回测假设暴露出来，而不是只展示一条漂亮曲线。

### 1.2 对 GRP 理念的适度映射

Principal 的历史公开产品材料将 GRP 描述为专有股票排名框架，用于系统覆盖大量公司，并依据区域行业同业对公司基本面特征进行排名，从而缩小分析师的重点研究范围。Mini-GRP 对这一理念做了三个层面的简化映射：

| 公开理念 | Mini-GRP 实现 | 有意保留的边界 |
|---|---|---|
| 扩大研究覆盖面 | 对统一股票池进行批量因子计算 | 不声称覆盖机构级全球股票宇宙 |
| 相对同业比较 | 市场内标准化、行业内排名 | 未实现行业专属模型和独立权重 |
| 量化筛选支持基本面研究 | 输出 Top-N 与维度分解 | 不自动替代分析师尽调和投资决策 |
| 基本面变化与估值 | Value、Quality、Growth | 依赖数据源的字段质量和时点完整性 |
| 预期变化 | SUE、EPS 修正、评级修正 | 无提供商数据时跳过，不再随机填充 |

因此，更准确的项目定位是“受 GRP 公开理念启发的 Mini 研究原型”，而不是“复刻 Principal 内部系统”。

### 1.3 核心研究问题

项目围绕一个稳定的问题展开：在有限数据、有限算力和有限开发时间下，如何构建一个可解释、可扩展并能支持人工研究的股票初筛系统？

这个问题进一步分解为五个工程问题：

1. 如何对齐 A 股和美股等市场的字段、代码、行业和货币信息？
2. 如何处理异常值、缺失值、因子方向和不同量纲？
3. 如何在部分维度不可用时保持评分逻辑一致？
4. 如何避免回测中的未来函数、日期错位和错误的成本核算？
5. 如何让演示结果、真实数据结果和失败状态在界面与输出中一眼可区分？

## 2. 版本演进与完整过程

以下时间线根据旧版构建报告、v3.2 技术报告、当前代码结构和文件版本说明重建。由于早期仓库缺少完整的版本标签与提交记录，时间线应理解为“功能演进脉络”，不是严格的逐提交审计。

### 2.1 v1.x：A 股四模块原型

最初版本聚焦于最小可运行闭环：

- 数据层：akshare 或模拟数据。
- 因子层：16 个因子，覆盖价值、质量、增长和动量。
- 评分层：四维度加权，早期权重为 30%/30%/20%/20%。
- 展示层：Top 20、静态图表和 Excel 报告。

这一阶段解决的是“能否把面试中的投资框架真正跑起来”。其价值在于模块化和可解释，但对数据时点、回测、数据来源和部署的处理仍然较轻。

### 2.2 v2.x：评分与报告扩展

第二阶段开始从单次选股扩展到分析流程：

- 增加行业内排名和更完整的维度分解。
- 增加静态图表、行业统计和导出报告。
- 引入 Walk-forward 回测概念与绩效指标框架。
- 强调“量化筛选后由人工做基本面验证”的工作方式。

这一阶段形成了项目叙事，但早期回测更接近流程演示，还没有可靠处理点时数据、基准、交易成本和调仓日期。

### 2.3 v3.0：统一数据与跨市场方向

v3.0 的主要变化是数据层扩展：

- A 股尝试 Tushare Pro 与 akshare。
- 美股尝试 Alpha Vantage 与 yfinance。
- 引入统一字段映射和 SQLite 缓存。
- 增加 `market` 与 `currency` 等跨市场元数据。
- 设计 CN、US、HK、JP、KR 的统一入口。

这一阶段的架构方向正确，但实现中存在一个重要风险：当真实数据不可用时自动回退模拟数据，而输出端没有足够清楚地区分两者。这会让“系统始终有结果”演变成“系统可能给出看似真实的结果”。

### 2.4 v3.1：ML 作为辅助层

v3.1 将 ML 定位为线性因子模型的 Overlay，而不是替代主模型。代码中包含：

- 因子有效性验证与筛选。
- XGBoost 非线性评分实验。
- 市场环境识别。
- 多臂老虎机动态权重实验。
- Purged CV、SHAP 等研究性模块。

这一方向体现了正确的工程克制：基线模型保持可解释，ML 仅用于识别有限的非线性交互。但当前仓库中的 ML 部分仍属于实验模块，缺少真实、严格的时间序列训练数据和完整集成测试，因此 v3.3 没有把它放进默认生产路径。

### 2.5 v3.2：模块化、文档化与 Streamlit 部署

v3.2 增加三项预期差因子，将模型升级为 19 因子、5 维度，并形成较完整的技术报告、概览和部署指南。Streamlit 页面用于展示选股、雷达图和回测结果。

但审查发现，v3.2 的“文档状态”和“远程代码状态”不一致：

- 远程主分支只发布了少量入口文件，完整模块主要留在本地目录。
- `app.py` 存在未闭合字符串，无法通过语法编译。
- `streamlit_app.py` 内嵌了另一套因子与回测逻辑，和模块代码发生分叉。
- Pandas 版本下 `freq='ME'` 直接报错。
- 命令行输出列选择包含重复 `composite_score`，导致行值变成 Series 并报格式化错误。
- 回测调仓日使用月末，而模拟快照使用月初，导致找不到数据并返回零结果。

这说明 v3.2 已经有较强的产品外观和文档表达，但还需要一次以可靠性为中心的收敛。

### 2.6 v3.3：可靠性、可审计性与仓库治理

本次 v3.3 的工作原则是：先保证结果的身份和计算路径可信，再扩展功能。

具体过程为：

1. 从远程 `main` 创建独立改进分支，避免覆盖本地历史和远程已有提交。
2. 将本地完整模块代码纳入工作分支，排除缓存、旧输出和临时文件。
3. 重写回测核心，修复日期、收益、基准、成本和年化计算。
4. 统一 `app.py`、`streamlit_app.py` 和 `core.main` 的领域逻辑。
5. 增加研究模式与演示模式，禁止研究模式使用模拟回退。
6. 增加数据来源与覆盖率元数据。
7. 修正跨市场标准化与不可用因子的处理。
8. 增加依赖分层、自动测试、CI、Docker 和部署说明。
9. 运行命令行和浏览器端到端验证。
10. 重新整理项目报告，使其反映“实际已验证能力”，而不是只描述设计愿景。

## 3. v3.3 系统架构

### 3.1 分层结构

| 层级 | 主要目录 | 责任 | 关键输出 |
|---|---|---|---|
| 数据层 | `data/` | 获取、映射、缓存、来源标记 | 统一股票截面 DataFrame |
| 因子层 | `core/factor_engine.py` | 缺失处理、缩尾、标准化、方向调整 | `{factor}_z` 与维度原始得分 |
| 评分层 | `core/scoring_engine.py` | 维度标准化、动态权重、百分位、行业排名 | `composite_score`、`industry_rank` |
| 回测层 | `backtest/engine.py` | 点时快照、持有期收益、换手成本、基准与指标 | 逐期结果、NAV、持仓与指标 |
| 入口层 | `core/main.py`、`streamlit_app.py` | 参数、模式、编排、交互、输出 | CLI 和 Web 应用 |
| 实验层 | `ml/`、`analytics/` | ML 与研究工具 | 非默认的研究性功能 |

### 3.2 单一逻辑来源

v3.2 的网页文件内嵌了一套独立的因子和回测实现，导致同一按钮在不同入口下可能执行不同算法。v3.3 将网页改为调用核心模块：

- 选股统一调用 `score_universe()`。
- 回测统一调用 `backtest.engine`。
- `app.py` 仅作为兼容包装器调用 `streamlit_app.main()`。
- CLI 与网页共享模拟数据生成器。

这一调整降低了重复代码，也让测试真正覆盖用户界面背后的核心行为。

### 3.3 运行模式

系统现在有两个明确的数据模式：

**Synthetic demo**

- 使用固定随机种子的合成数据。
- 适合演示、回归测试和界面验证。
- 所有输出标记 `data_source=synthetic_demo`、`is_mock=True`。
- 回测收益和因子信号存在人为设定的适度关系，仅用于验证流程。

**Research data**

- 使用 Tushare、akshare、Alpha Vantage 或 yfinance 等真实提供商。
- 数据不可用时抛出明确错误。
- 不生成随机预期因子。
- 需要用户审查字段覆盖率、数据日期和提供商限制。

## 4. 数据层与数据治理

### 4.1 统一字段

统一数据表至少包含三类字段：

1. 标识字段：`code`、`name`、`sw_industry_name`、`market`、`currency`。
2. 因子字段：19 个候选因子。
3. 治理字段：`data_source`、`is_mock`、`as_of_date`、`factor_coverage`、`expectation_source`。

治理字段是 v3.3 的关键改进。对于任何选股结果，用户都可以先回答：数据来自哪里、是否模拟、截至哪一天、因子覆盖是否完整、预期数据是否真实存在。

### 4.2 数据优先级

| 市场 | 首选 | 备选 | 演示回退 |
|---|---|---|---|
| A 股 | Tushare Pro | akshare | 仅 demo 模式允许 |
| 美股 | Alpha Vantage | yfinance | 仅 demo 模式允许 |
| 港股/日股/韩股 | 预留 Yahoo 适配方向 | 尚需完善 | 仅 demo 模式允许 |

### 4.3 缺失数据原则

旧版本为了维持五维输出，会对缺少分析师预期数据的市场生成随机 SUE、EPS 修正和评级修正。这种做法适合 UI 占位，但不适合研究输出。v3.3 的处理原则是：

- 列不存在、全空或没有横截面差异时，该因子不可用。
- 不可用因子不会生成标准化列。
- 整个维度不可用时，维度得分设为 NaN。
- 综合评分只使用活跃维度，并重新归一化权重。

这使得模型宁可承认信息缺失，也不会用随机值维持表面完整。

### 4.4 缓存边界

缓存用于降低接口调用频率，但缓存不能改变数据身份。v3.3 在读取缓存时检查 `is_mock`，研究模式不会接受历史上以真实源键保存的模拟数据。后续仍建议把缓存键进一步扩展为：提供商、市场、股票集合、字段版本、请求日期和模式的组合，并为数据模式设置独立命名空间。

## 5. 因子模型

### 5.1 因子定义

| 维度 | 因子 | 方向 | 经济含义 |
|---|---|---:|---|
| Value | PE TTM | -1 | 盈利对应的估值水平 |
| Value | PB LF | -1 | 净资产对应的估值水平 |
| Value | PS TTM | -1 | 收入对应的估值水平 |
| Value | EV/EBITDA | -1 | 企业价值相对经营利润 |
| Value | Dividend Yield | +1 | 现金分红回报 |
| Quality | ROE | +1 | 股东资本盈利能力 |
| Quality | ROA | +1 | 总资产盈利效率 |
| Quality | Gross Margin | +1 | 产品与商业模式盈利空间 |
| Quality | Net Margin | +1 | 最终利润转化能力 |
| Quality | Debt/Equity | -1 | 财务杠杆与偿债压力 |
| Growth | Revenue YoY | +1 | 收入扩张速度 |
| Growth | Profit YoY | +1 | 盈利扩张速度 |
| Growth | FCF Yield | +1 | 自由现金流相对估值 |
| Momentum | Return 1M | +1 | 短期价格趋势 |
| Momentum | Return 3M | +1 | 中短期价格趋势 |
| Momentum | Return 12M | +1 | 中期价格趋势 |
| Expectation | SUE | +1 | 实际盈利相对预期的惊喜程度 |
| Expectation | EPS Revision | +1 | 分析师盈利预期修正方向 |
| Expectation | Rating Revision | +1 | 分析师评级变化方向 |

### 5.2 预处理流程

每个可用因子的处理顺序为：

1. 数值转换与有效性检查。
2. 缺失值优先按市场与行业均值填充，再按市场均值和全局均值补充。
3. 5%/95% 分位缩尾，降低极端值影响。
4. 在各市场内部进行 Z-Score 标准化。
5. 根据因子方向乘以 +1 或 -1，使得高分统一表示更优。
6. 可选行业中性化。

标准化公式为：

`z_i = (x_i - mean(x)) / std(x)`

对低值更优的估值与杠杆因子，最终得分为 `-z_i`。

### 5.3 为什么采用市场内标准化

不同市场的估值中枢、会计口径、行业结构和货币环境不同。如果直接把 A 股 PE 与美股 PE 放在同一分布中做 Z-Score，模型可能把“市场差异”误认为“公司差异”。v3.3 先在市场内部缩尾和标准化，再进行综合排名。这并不能解决所有跨市场可比性问题，但比全样本混合标准化更合理。

## 6. 评分与排名

### 6.1 维度聚合

维度得分是该维度下可用标准化因子的等权平均，再次进行横截面标准化。例如：

`Value = mean(PE_z, PB_z, PS_z, EV_EBITDA_z, DividendYield_z)`

默认权重为：

| 维度 | 权重 |
|---|---:|
| Value | 25% |
| Quality | 25% |
| Growth | 15% |
| Momentum | 15% |
| Expectation | 20% |

### 6.2 动态权重

若预期维度不可用，剩余权重按比例重新归一化，而不是把缺失预期得分当作零惩罚所有股票。设活跃维度集合为 A，则：

`normalized_weight_d = weight_d / sum(weight_j for j in A)`

### 6.3 综合评分

原始综合分为各维度标准分的加权和：

`raw_score = sum(normalized_weight_d * dimension_score_d)`

随后转换为 0 至 100 的百分位：

`composite_score = percentile_rank(raw_score) * 100`

百分位便于解释“位于当前股票池的前多少比例”，但它不是预期收益率，也不应被解读为成功概率。

### 6.4 行业内排名

`industry_rank` 在市场与行业组合内部计算。这样可以同时查看全股票池的综合排序，以及一家公司在本地行业同业中的相对位置。当前实现仍使用统一权重，尚未达到真正的行业特定模型。

## 7. Walk-forward 回测框架

### 7.1 点时数据契约

研究回测需要两类输入：

- 调仓日可见的股票因子快照，包含 `period`、股票标识、行业和因子。
- 与每个调仓日对应的持有期价格窗口。

每一期只使用当期快照评分，选出 Top-N，随后用下一调仓日前的价格计算持有期收益。

### 7.2 组合、基准与成本

组合采用等权持有。基准为当期可投资股票池的等权收益，不再使用随机基准。

交易成本基于目标权重和上一期权重之间的绝对变化：

`traded_notional = sum(abs(target_weight - previous_weight))`

`cost = traded_notional * one_way_transaction_cost`

该计算同时覆盖卖出和买入，不再额外错误乘以 100 或重复乘以双边系数。

### 7.3 关键指标

系统输出：

- 总收益与年化收益。
- 基准收益与年化基准收益。
- 总超额与年化超额。
- 超额收益口径 Sharpe。
- 最大回撤与 Calmar。
- 超额胜率。
- Rank IC 与 ICIR。
- 平均换手、交易次数和累计交易成本。

月度与季度回测使用不同的年化期数，不再统一假设 12 期。

### 7.4 硬性失败门槛

以下情况会抛出错误，而不是返回全零结果：

- 没有点时快照。
- 没有对应价格窗口。
- 选股结果为空。
- 持仓没有有效收益。
- NAV 出现 NaN 或无穷值。
- 整个回测没有交易或持仓。

这项改变非常重要，因为“失败但输出 0”比直接报错更危险，它会把数据管道问题伪装成策略表现。

## 8. v3.2 审查发现与修复

| 问题 | 影响 | v3.3 处理 |
|---|---|---|
| `app.py` 字符串未闭合 | 入口无法编译 | 重建为兼容包装器 |
| Streamlit 内嵌另一套模型 | 三个入口结果可能不一致 | UI 调用核心模块 |
| Pandas `ME` 频率不兼容 | 回测按钮直接崩溃 | 使用 `MonthEnd`/`QuarterEnd` offset |
| `composite_score` 重复选列 | 行值成为 Series，CLI 报错 | 使用显式维度列清单 |
| 月初快照匹配月末调仓日 | 每期无数据 | 统一调仓日期与快照键 |
| 回测导入旧模块路径 | 异常被捕获后返回零 | 使用包路径并移除静默吞错 |
| 交易成本量纲错误 | 成本被放大或重复计算 | 权重差额口径统一计算 |
| 基准收益随机生成 | 超额收益没有经济含义 | 使用当期股票池等权基准 |
| 累计超额简单相加 | 长期超额计算错误 | 使用组合 NAV / 基准 NAV |
| 所有频率按月年化 | 季度指标错误 | 根据频率设置 12 或 4 |
| 真实数据缺失时随机补预期 | 输出身份不清 | 研究模式跳过缺失维度 |
| Mock 可能进入真实缓存键 | 后续读取误判来源 | 缓存读取检查 `is_mock` |
| 测试目录为空 | 回归风险高 | 新增 7 项测试与 CI |
| 远程仓库内容不完整 | 面试官看不到真实工程量 | 纳入完整模块与文档 |

## 10. 部署、产物与工程治理

### 10.1 依赖分层

- `requirements.txt`：网页、因子、演示和回测基础依赖。
- `requirements-dev.txt`：测试和覆盖率。
- `requirements-research.txt`：真实数据提供商适配器。

这种拆分避免 Streamlit 演示部署被重型或不稳定的数据包拖累，也让真实研究能力成为显式选择。

### 10.2 自动化

GitHub Actions 在 push 和 pull request 时执行：

1. 安装开发依赖。
2. 编译全部 Python 文件。
3. 运行测试。
4. 运行演示选股烟雾测试。
5. 运行演示回测烟雾测试。

### 10.3 输出契约

选股输出完整股票池、Top Picks 和运行清单。回测输出逐期结果、权益序列、持仓、指标、清单和权益图。清单记录生成时间、数据模式、期间数、交易数和文件列表，便于后续复核。

### 10.4 部署方式

项目支持：

- 本地 Python + Streamlit。
- Streamlit Community Cloud。
- Docker 容器。

默认部署只安装基础依赖并进入合成演示模式。需要真实数据时，额外安装研究依赖并通过环境变量或 Streamlit Secrets 配置密钥。

## 11. 当前限制与风险

### 11.1 数据风险

1. 免费接口可能限流、改字段或暂时不可用。
2. 财务数据需要严格的公告日和可得日，而不是简单使用报告期。
3. 股票池需要历史成分和退市股票，避免幸存者偏差。
4. 价格需要复权、停牌、涨跌停和公司行动处理。
5. 分析师预期数据的覆盖与授权是预期差维度的核心约束。

### 11.2 模型风险

1. 权重目前是研究假设，不是经过严格样本外优化的参数。
2. 行业内排名不等于行业专属因子模型。
3. 因子可能高度相关，等权聚合不能完全解决共线性。
4. 百分位评分只表达相对位置，不表达绝对收益。
5. ML 模块容易在小样本金融数据中发生过拟合。

### 11.3 回测风险

1. 合成演示不能用于判断可投资性。
2. 当前 CLI 不提供真实历史点时数据下载，研究回测必须由调用方提供快照和价格窗口。
3. 等权基准只是内部诊断基准，真实产品应使用可投资指数和一致的交易日历。
4. 成本模型尚未包含冲击成本、买卖价差、容量和成交限制。

### 11.4 合规与表达风险

旧报告中存在把设计目标写成已实现能力、把模拟回测写得接近历史验证、以及引用可能过时机构数据的问题。v3.3 报告采用以下表达纪律：

- 合成结果始终标记为合成演示。
- “支持某市场”区分为接口预留、可获取当前快照和已完成历史验证。
- 不使用未经核实的最新 AUM、组织架构或人员信息支撑项目价值。
- 不把项目称为机构系统的完整复刻。

## 12. 后续路线图

### P0：真实研究闭环

1. 建立点时数据仓库，记录公告日、数据可得日和版本。
2. 引入历史股票池、退市样本和公司行动调整。
3. 接入真实分析师预期历史，而不是当前快照。
4. 增加数据质量门槛：最小覆盖率、异常率、重复率和新鲜度。
5. 为真实研究回测增加独立集成测试。

### P1：组合与风险

1. 增加行业、单股、市场和风格暴露约束。
2. 从 Top-N 等权扩展到风险预算或优化权重。
3. 加入换手上限、流动性、涨跌停和容量约束。
4. 增加年度归因、行业归因、因子归因和交易成本归因。
5. 使用可投资指数作为市场基准。

### P2：模型研究

1. 对每个因子进行 Rank IC、分层收益、单调性和稳定性检验。
2. 使用 Purged Walk-forward CV 验证 ML Overlay。
3. 加入因子相关性、VIF、正交化或聚类降维。
4. 建立 Champion-Challenger 流程和模型版本登记。
5. 只有在样本外结果稳定后，才让 ML 进入默认路径。

### P3：产品化

1. 增加用户认证、任务队列和运行历史。
2. 将数据、模型、结果和报告拆成可追踪任务。
3. 增加监控、告警、数据血缘和审计日志。
4. 在纸面交易环境验证后，再考虑券商接口。

## 13. 面试展示建议

建议用 8 至 10 分钟完成演示：

1. 用 45 秒说明问题：如何把广泛股票池压缩为分析师可研究的候选集合。
2. 用 60 秒说明架构：数据、因子、评分、回测、界面。
3. 用 90 秒展示五维因子与动态权重。
4. 用 90 秒运行合成选股，解释数据来源标签和因子覆盖率。
5. 用 90 秒展示回测，但明确它是流程检查，不是业绩证明。
6. 用 90 秒讲最重要的工程改进：禁止静默 Mock、点时数据契约、成本和基准。
7. 用 60 秒讲下一步：真实历史预期、幸存者偏差、组合约束。

最有价值的面试叙事不是“我做了一个很复杂的模型”，而是：

> 我先把投资理念转成一个可解释原型，然后在迭代中发现，量化系统最危险的不是模型不够复杂，而是数据身份不清、失败被隐藏、回测假设不一致。因此我把最新版本的重点放在可审计性和可靠性上。

这段叙事同时体现投资理解、工程判断、诚实边界和迭代能力。

## 14. 结论

Mini-GRP 已经从一个面试展示原型，发展为结构完整的个人量化研究项目。v3.3 的意义不在于因子数量继续增加，而在于项目终于形成一致的运行入口、明确的数据身份、可重复的回测、自动化验证和与实际实现一致的文档。

它目前最适合承担三种角色：

- 作为面试作品，展示从投资思想到工程实现的完整能力。
- 作为研究脚手架，承接未来的真实点时数据与组合约束。
- 作为学习项目，演示量化系统中数据、模型、回测、部署和治理之间的关系。

下一阶段的关键不是增加更多算法，而是获得合规、可靠、历史可追溯的数据，并在严格样本外框架下证明因子与组合的稳定性。
